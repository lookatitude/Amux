cmux to Amux parity gap audit#

Amux has the daemon-first terminal foundation. It does not yet have the browser, rich desktop workspace shell, SSH runtime, agent-session depth, or Ghostty-class terminal polish that define full cmux product parity.

Bottom line#

Amux is a strong clean-room implementation of cmux's core runtime ideas, not a full port. It already has a Go authority, PTY supervision, versioned local protocol, attach/replay/input leases, snapshots, notifications, project-scoped hook trust, context discovery, CLI flows, and a real Bubble Tea split-tree TUI.

There are three distinct finish lines:

  1. Approved Linux MVP: close the reopened backend, security, and release findings, then rerun the complete T6 release-evidence lane.
  2. Terminal-centric parity: add terminal fidelity, rich workspace/sidebar UX, deep agent integrations, broad automation, and notification polish.
  3. Full product parity: add the embedded scriptable browser, SSH remotes, native desktop shell, macOS/Windows, and companion systems.

Missing before the Linux MVP ships#

Close reopened T2, T3, and T4 findings

The first integrated T6 pass exposed defects missed by lane-local suites. The final candidate still needs atomic trust transitions including audit, one-lock replay snapshots, a deterministic PTY drain barrier, truthful Linux race evidence, and a clean pinned GoReleaser/TMPDIR/package path. At this refresh the trust-store interface is mid-integration, so the live tree is not a releasable baseline.

Rerun T6 release evidence

  • Traceability, property, fuzz, golden, protocol, VT, PTY, persistence, and security suites.
  • All 20 CLI flows against installed binaries and real-terminal eight-pane acceptance.
  • Fault injection for frames, disks, snapshots, SQLite, daemon death, hook limits, and reconnects.
  • Arch reference performance, a blocking 30-minute soak, and a nightly 8-hour soak.
  • Clean Arch and Ubuntu package installs, checksums, SBOM/provenance, rollback, and release security review.

Cross-compilation from macOS is not Linux runtime proof. cgroup-v2 containment, descriptor-bound launch, AUR installation, D-Bus notification delivery, PTY behavior, and soak claims require Linux execution.

The first candidate did complete a 30-minute Arch-container soak with no gaps or orphans and exercised install/daemon/CLI smoke on Arch amd64, Ubuntu amd64, and Ubuntu arm64. Those are useful harness proofs, but the receipts must be regenerated after the reopened fixes.

Missing for terminal-centric cmux parity#

Ghostty-class terminal experience#

The pure-Go VT engine and Bubble Tea renderer fit the Linux MVP. They do not reproduce cmux's GPU-accelerated Ghostty experience.

  • GPU rendering and Ghostty configuration compatibility.
  • Mature selection, copy semantics, terminal/directory search, IME, and complex input.
  • Hyperlinks, file drops, image protocols, uploads, fonts, themes, scroll controls, and clear-scrollback actions.
  • Long-tail VT fidelity, resource inspection, memory-pressure handling, and renderer lifecycle controls.

Rich workspace and sidebar product#

  • Multiple desktop windows, vertical workspace tabs, richer horizontal surface tabs, and window restore.
  • Workspace groups, anchors, pinning, colors/icons, reorder, drag/drop, and richer pane move/swap/join/zoom operations.
  • Linked PRs, listening ports, progress, logs, todos, auto-naming, and task manager.
  • File explorer, global search, command palette, right sidebar modes, project custom commands, Dock, and safe custom sidebars.

Agent integration depth#

Amux has secure hooks and initial Claude/Codex attention parsing. It still lacks the broader agent product:

  • Provider installer/adaptor catalog, durable native session IDs, resume and conversation fork.
  • Agent hibernation, focus-triggered resume, Feed approval/question/plan workflows, and team launchers.
  • Custom detection/resume templates, auto-naming, richer agent state, and policy-driven notifications.

Automation and compatibility#

  • The larger command vocabulary for windows, groups, todos, Dock, browser, SSH, themes, diagnostics, and custom sidebars.
  • Partial tmux compatibility, published multi-language SDKs, and a WebSocket gateway.
  • Deep links and native file/service integrations.
  • cmux CLI/config/socket compatibility, if explicitly chosen. It is intentionally not promised today.

Missing for full cmux product parity#

Embedded, agent-scriptable browser#

This is the largest missing differentiator. Full parity means browser surfaces in the same graph, with omnibar/navigation/history, profiles and browser import, downloads/uploads/devtools, and a broad automation API for DOM/accessibility snapshots, refs, input, JavaScript, waits, screenshots, storage, network, and proxy control.

SSH and remote workspaces#

cmux has remote daemon deployment, persistent remote PTYs, reconnect, authenticated CLI relay, browser egress, scp uploads, SSH-agent controls, and resize coordination. Amux is owner-only and local-only. Remote support needs a new protocol and threat model, not an exposed local socket.

Native desktop shell and platforms#

  • Desktop windows, menus, global shortcuts, tray, drag/drop, clipboard, IME, accessibility, and native focus.
  • macOS runtime adapters, signed/notarized packaging, and native notification/update behavior.
  • Windows ConPTY, named pipes, ACL/peer validation, process containment, installer, and update support.
  • Linux GUI packaging with Wayland/X11 validation and stable/nightly updates.
  • Browser/window/sidebar restore, agent resume bindings, crash diagnostics, telemetry policy, and localization.

Cloud VMs, mobile clients, Agent Chat, and freeform canvas are additional repository surfaces, but several are gated or experimental in cmux. They should be optional parity tiers, not blockers for a credible desktop release.

The refreshed cmux main snapshot also contains password-authorized control sockets, remote tmux mirroring, a native diff sidecar/viewer, browser viewport and automation watchdog work, Iroh-backed mobile connectivity, and richer iOS terminal behavior. Mirroring every directory on main is therefore a moving-target product program, not a finite desktop port.

Architecture decision#

Keep Go as the sole daemon, state, and security authority. Do not rewrite the core to chase GUI parity. Add clients around the existing protocol:

Go daemon authority
  → local Unix socket / future authenticated remote protocol
  → Bubble Tea client (Linux-first)
  → future desktop client (Qt/QML if browser parity is mandatory)
  → future authenticated web/remote gateway

The first major post-release decision is the browser tier. Choose Qt WebEngine if embedded browser parity is a core promise. Otherwise, validate the workflow with external Chromium/CDP and keep the terminal product lean.
