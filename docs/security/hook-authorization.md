# Hook authorization contract (T2-security, frozen pre-implementation)

Status: frozen 2026-07-15 (run-8c2926bb-2ebe-4166-8b77-84c0de3e12c0, lane T2-security).
This is the normative fail-closed contract T4 backend implements and T6 QA
verifies. Each requirement has a stable ID (`HA-n`). The machine-checkable
expected outcomes live in `testdata/security/trust-matrix.json` (generated by
`internal/securitytest`); the deterministic adversarial fixtures that pin the
concurrency and race requirements live in `testdata/security/fixtures/` and run
through `securitytest.RunConformance`.

Terminology is the approved spec's ("Project identity and trust boundary",
"Hook trust and execution", "Linearizable hook launch contract") and PRD F9.
Frozen upstream contracts consumed here, never redefined: `platform.FSIdentity`,
`platform.ComputeProjectKey`, `platform.DescriptorLaunch`, `platform.Containment`,
`platform.PeerCredentials`, `platform.Clock`, the `api/v1` error taxonomy, and
the ADR-0005 authority table (trust state is SQLite-only, control-actor-owned).

## 1. Project identity (HA-2)

- **HA-2a** A project is identified by the canonical `realpath` of an explicitly
  selected root plus that root's filesystem identity (`st_dev`, `st_ino`). The
  durable key is `platform.ComputeProjectKey` — the hex SHA-256 of the
  length-prefixed tuple (`"amux-project-v1"`, realpath, dev, ino). No other
  derivation is permitted.
- **HA-2b** Canonicalization and identity capture are one atomic observation:
  the implementation must resolve the path and capture (dev, ino) from the same
  opened directory descriptor (or verify post-resolve stat equality against the
  descriptor) so no time-of-check/time-of-use gap exists between "which
  directory" and "which identity". Path-string comparison alone is forbidden.
- **HA-2c** Moving, replacing, remounting, or recreating a root changes the
  tuple and therefore the key; all trust and grants bound to the old key are
  unreachable from the new object (matrix rows `project.replaced-root`,
  `project.remounted-root`).
- **HA-2d** A Git-backed pane may *discover* a candidate root (worktree root);
  a non-Git pane has no project until explicit registration. Discovery never
  creates trust: cwd alone never establishes a trust boundary.
- **HA-2e (replacement validation — G-lane F2 amendment, 2026-07-17)** The
  HA-2a key definition is unchanged and remains the sole durable identity.
  However, some filesystems (overlayfs — the container deployment class)
  deterministically reuse a directory's `(st_dev, st_ino)` across
  `rm -rf root && mkdir root`, so the key alone cannot uphold HA-2c there. A
  SEPARATELY PERSISTED replacement-validation discriminator
  (`platform.ReplacementValidator`, stored beside the project row) closes
  this: on Linux it is derived from the root's `statx` birth time
  (`STATX_BTIME`) — a kernel identity surface that changes exactly when the
  object at the path is recreated and never under ordinary child
  create/write/remove; on the Darwin author host, from `st_birthtimespec`.
  The discriminator is (re)checked at every registration, at restart
  rehydration of persisted trust (`TrustStore.LoadProject` feeds the actor
  only through this recheck), and at the pre-launch root identity recheck.
  A mismatch — or an ABSENT persisted value (pre-amendment row) — is an
  identity replacement: approved trust is invalidated as a system revocation
  (monotonic epoch bump, grants deactivated, `project_revoked` audit record
  carrying the `project_trust_required` code, revoke listeners notified).
  Capability is explicit and fail-closed: a kernel/filesystem that cannot
  provide the surface (`statx` masks out `STATX_BTIME`, ENOSYS, zero birth
  time, unsupported platform) yields a typed, audited
  `project_trust_required` denial at registration — trust reuse is denied,
  never guessed. Documented conservative edge: on overlayfs, the first
  copy-up of a root that existed only in a lower layer refreshes the upper
  inode's birth time, so trust granted before that first modification is
  invalidated once and must be re-approved (fail-closed direction).
  Mount identity (`stx_mnt_id`) was rejected as the surface because it
  changes across reboots of an unchanged root; inode generation
  (`FS_IOC_GETVERSION`) because it is privileged/unsupported on common
  overlay stacks.

## 2. Trust state machine (HA-3, HA-4)

Per project key, owned globally by the daemon control actor, persisted SQLite-only:

```
unregistered --register/discover--> registered
registered   --opt_in+approve-----> trusted(epoch=E)      [operator action, explicit]
registered   --deny---------------> denied                [recorded, config stays ignored]
trusted(E)   --revoke-------------> revoked(epoch=E+1)    [grants -> inactive, queue canceled]
trusted(E)   --identity change----> (old key orphaned; new object = unregistered)
revoked(E+1) --reapprove----------> trusted(epoch=E+2)    [prior grants stay inactive]
denied       --approve------------> trusted(new epoch)
```

- **HA-3a** Hook configuration (`.amux/hooks.jsonc`, the only permitted config
  path, project-relative and fixed) is not read, parsed, or diagnosed before the
  project is in `trusted`. (`HA-1` in the threat model: config ignored pre-opt-in.)
- **HA-3b** States other than `trusted` yield `project_trust_required` for any
  hook activation, within the HA-22 gate, with zero processes created and no
  entry to the runnable set.
- **HA-4a (epoch monotonicity)** Each project carries a monotonic trust epoch,
  global across all sessions and daemon restarts. Every approve/revoke/reapprove
  transition strictly increases it. An epoch value is never reused and never
  decreases — including across snapshot restore of any generation (`HA-18`).
- **HA-4b** Epoch state is control-actor-serialized: concurrent trust
  transitions from different sessions observe a single total order.

## 3. Grant binding (HA-5..HA-9)

- **HA-5** Project trust alone authorizes nothing but config reading. Each hook
  additionally requires its own grant. Execution audit references both (project
  approval epoch + grant ID).
- **HA-6** A grant binds ALL of: project key, project epoch at approval, hook ID,
  executable absolute path, executable SHA-256, config (`hooks.jsonc`) SHA-256,
  allowed event set, cwd scope, environment-key allowlist, timeout, output cap.
- **HA-7 (invalidation)** A change to any HA-6 field observed at validation time
  — executable bytes, config bytes, event set, cwd scope, env allowlist,
  timeout, cap — makes the grant stale: activation fails `hook_grant_stale`,
  the grant is marked inactive (history retained), and fresh approval is
  required. A grant whose bound epoch is not the project's current epoch is
  stale for the same reason.
- **HA-8 (bounds)** Timeout defaults to 2 000 ms; the maximum grantable timeout
  is 30 000 ms; output (stdout+stderr combined) is capped at 1 MiB
  (1 048 576 bytes). A config or grant request exceeding a bound fails
  `invalid_argument` at grant time — never clamped silently. The environment
  allowlist defaults to empty; only listed keys are injected, values sourced
  from the daemon's explicit non-secret store, never ambient inheritance.
- **HA-9 (cwd scopes)** Cwd access is denied by default; hooks run in an
  Amux-owned scratch directory. A grant selects exactly one scope:
  - `fixed`: one absolute path granted verbatim; the daemon validates the
    resolved path is that object (identity check, not string check).
  - `workspace-primary`: resolves only when the workspace has exactly one
    explicitly configured primary root AND that root's *current* project
    identity equals the grant's project key. Absent, ambiguous (≠1), replaced,
    or foreign root ⇒ `scope_denied` before launch.
  - `pane`: resolves only when the target pane's discovered/registered project
    identity equals the grant's project key. Cross-project, unregistered, or
    identity-changed panes ⇒ `scope_denied` before launch.
  An activation whose requested event is outside the granted event set, or that
  requests env keys outside the allowlist, fails `scope_denied` (grant is
  intact but the requested authority exceeds it).

## 4. Descriptor-bound launch (HA-10..HA-13)

Path-only revalidation is explicitly insufficient (PRD F9). The launch pipeline
must bind validation and execution to the same opened objects:

- **HA-10** The executable and config are opened via
  `platform.DescriptorLaunch.OpenBound` semantics: resolution relative to a
  trusted directory descriptor with no symlink and no magic-link traversal.
- **HA-11** Digests are computed from the *opened descriptors'* bytes and
  compared to the grant's bound digests. The subsequently launched object is
  that exact descriptor (`LaunchBound`, `fexecve`/`execveat` semantics) — a
  symlink swap, rename, byte replacement, or root replacement between check and
  exec cannot substitute a different object. Pinned by `races.symlink-swap`,
  `races.rename-swap`, `races.exec-byte-replace`, `races.config-byte-replace`,
  `races.project-root-replace`: the approved object executes or launch fails
  closed `hook_grant_stale`; the substituted object never executes.
- **HA-12** The opened executable must be a regular file (descriptor type
  check); FIFOs, devices, directories, and unsupported executable forms fail
  closed `invalid_argument`.
- **HA-13** Cwd-scope resolution (HA-9) is likewise identity-bound: the
  resolved directory's (dev, ino) is captured with the resolution and handed to
  launch; a directory replaced after resolution fails the pre-launch identity
  recheck ⇒ `scope_denied`.

## 5. Linearizable launch/revoke (HA-14..HA-17)

The T1-frozen boundary (`AuthorityControlActor`) serializes all trust
transitions and launch authorizations for a project. Consuming that design, the
**final pre-spawn authorization point** is: inside the control actor's
serialized section, after descriptor open + digest verification, the atomic
check `state == trusted ∧ grant.active ∧ grant.epoch == current epoch ∧ scope
resolved ∧ bounds valid`, immediately followed by `LaunchBound` with no
awaitable step between validation and spawn.

- **HA-14 (revoke-first)** If a revoke linearizes before that point, the
  activation fails closed and **zero** hook children are created — pinned by
  `timing.revoke-first` under a deterministic barrier holding the activation at
  `before-final-validation`.
- **HA-15 (launch-first)** If launch linearizes first, the invocation is
  in-flight: revocation delivers terminate immediately and escalates to
  kill-tree at exactly the 2 000 ms boundary (fake-clock deterministic), with
  terminate, escalation, and exit all audit-visible — pinned by
  `timing.launch-first`.
- **HA-16** Termination targets the containment handle (whole descendant tree),
  never a raced PID.
- **HA-17** Hook children receive only: the granted argv (typed
  executable-plus-arguments, never shell-interpolated strings), the allowlisted
  env, the resolved cwd (or scratch), and a minimal descriptor set
  (close-on-exec everywhere else).

## 6. Revocation and restore (HA-18..HA-21)

- **HA-18** Revocation, from any session, (a) increments the epoch, (b) cancels
  every queued same-project invocation across **all** sessions within 250 ms,
  (c) prevents any later launch under the old epoch, (d) marks grants inactive
  while retaining full grant + audit history, and (e) requires BOTH fresh
  project approval and fresh per-hook grants before any future launch — pinned
  by `timing.revoke-cancel`.
- **HA-19** Trust state (epochs, grants, audit) is SQLite-only and is never
  imported from a layout snapshot (ADR-0005). A restore of any generation —
  old layout, old notification export, old replay sidecar — must not decrease
  any epoch (`restore.epoch-monotonic`), must not reactivate any grant
  (`restore.grant-stays-inactive`), and must not erase or truncate inactive
  audit history (`restore.audit-retained`).
- **HA-20** No snapshot content can authorize a launch: authorization reads
  only live control-actor state (`restore.no-launch-authority`).
- **HA-21** A corrupt or partial trust store fails closed: hook subsystem
  disabled with an actionable diagnostic; core terminal/session behavior
  unaffected (subsystem is independently disableable end-to-end).

## 7. Timing gates (HA-22)

All gates are measured with the injectable `platform.Clock`; fixtures use a
deterministic fake clock, integrated tests measure wall time.

| Gate | Bound | Error/effect | Fixture |
|------|-------|--------------|---------|
| Absent/insufficient trust answer | ≤ 250 ms | `project_trust_required`, zero children, not runnable | `timing.absent-trust` |
| Cross-session revocation cancel | ≤ 250 ms | queued work canceled, audited | `timing.revoke-cancel` |
| Terminate→kill escalation | exactly at 2 000 ms after terminate | kill-tree, audited | `timing.launch-first` |
| Default hook timeout | 2 000 ms | terminate→kill path, `resource_exhausted` recorded in audit outcome | matrix `bounds.timeout-default` |
| Max grantable timeout | 30 000 ms | larger request ⇒ `invalid_argument` | matrix `bounds.timeout-exceeds-max` |
| Output cap | 1 048 576 bytes | truncation marker (redaction-safe), audited | matrix `bounds.output-cap`, `redaction.truncation` |

These constants are frozen in code as `securitytest.Gates` and cross-checked
against `docs/security/readiness-manifest.json` by `TestGateConstantsMatchManifest`.

## 8. Fail-closed error codes

Only the frozen `api/v1` taxonomy is used; no new codes. Mapping (normative,
machine-pinned by the trust matrix):

| Condition | Code |
|-----------|------|
| Project unregistered / registered / denied / revoked / identity-changed root | `project_trust_required` |
| Project trusted, no grant for this hook | `hook_grant_required` |
| Grant exists but any HA-6 binding changed, or epoch mismatch, or digest mismatch at descriptor validation | `hook_grant_stale` |
| Cwd scope unresolvable / cross-project / event outside granted set / env key outside allowlist | `scope_denied` |
| Bound exceeded at grant/config time; non-regular-file executable | `invalid_argument` |
| Timeout/output-cap enforcement, queue/connection limits | `resource_exhausted` |
| Invariant violation inside the trust engine | `internal` (fail closed, never grant) |

Every failure is fail-closed: on any error, ambiguity, or invariant doubt the
activation is denied, zero processes are created, and the denial is audited.
Noninteractive contexts receive the code plus actionable instructions
(which approval command to run); they never receive an interactive prompt.

## 9. Ownership

Security owns this contract, the matrix, and the fixtures. T4 backend
implements it (`internal/control` / `internal/hooks` per the PRD architecture)
behind the frozen platform seams and registers its system under test with
`securitytest.RunConformance`. Widening any grant authority (cwd, env, events),
weakening peer validation, or changing revocation guarantees requires the
orchestrator confirmation gate — these are contract changes, not
implementation details.
